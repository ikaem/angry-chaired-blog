WhatsApp Clone with React and Webpack: The Setup

16/09/2020
whatsapp react webpack

Part 1 of 6 in Whatsapp Clone with React and Webpack

We are going to build a React app that wants to be Whatsapp. Good luck with that, right? Yeah… We will use something called Socket.io to make messaging between people possible. And that is all we need to know for now.

See Project Information

On the more technical side, we will not be using create-react-app tool to set up a React app. We will do so on our own, by manually configuring Webpack, Babel, and ESLint.
This will create (actually, it already has created) tons of problems. We will have to google, and think, and find ways around those problems, to move on with the project.
This will pay off, though. We will learn how these tools actually work together to create a React app.

We will create both front and back end sides of the app. We will not have a database to store messages or user profiles (not yet), but we will have few JS objects that will be used for those purposes.

On frontend we will also use Tailwind to style the app (but we will not go over the styling part in this series), React Router to move between pages and components, and TypeScript, to write our code.
Backend will be simple: Node, Express, and Nodemon.

That’s it. We start with creating the app with Webpack and its related plugins.

Initialize the project

First we need to initialize our project. We can do that by creating a package.json file in the root folder of our project. For now, we will add two attributes in the object inside the file: name and version.

Next, we will install React packages: react and reactDOM.

We also need to install their TS types. We will do it as dev dependencies.

We move on. Let’s create three folders in the root of our project: src, build, and public.
We use src to create our project with TypeScript. The build folder will eventually be used to hold a the production build of the project.
Inside the public folder we put an index.html file. This will be the HTML document that in which our React app will live.

Few things about index.html:

  • script tag pointing to bundle.js in the same folder. At the moment, there is no bundle.js file in the public folder, but we will later configure the app to place all static files into the same folder, to make this path correct.
  • base html tag, with href attribute pointing to “/“. This tag specifies base URL for all realtive URLs in the document. We need it to be able to correctly redirect users to specific room URLs.
  • div container with the root id. This container will be used to insert the React app into index.html

Finally, we create an entry point to our React app. We call it index.tsx, and place it in the src folder. This file is the one that starts the whole application.

At the moment we are using index.tsx to insert a provisional App component into the DOM. Later, we will change this to have a proper component instead.

This project uses TypeScript only for type checking, and not for transpiling TS to JS. Later, we will configure Babel to do that.

Here is more info about each option:

  1. lib defines default libraries that TS will know for this project. In our case, TS will know about dom, dom.iterable, and esnext. This way, we can type check DOM objects, as well as the newest features of JS beloging to ESNext.
  2. allowJs allows JS files to be compiled too, in addition to TS ones.
  3. allowSyntheticDefaultImports enables type checking of default imports from modules with no default exports
  4. skipLibCheck that defines that we should skip type checking of all type declaration files - .d.ts files.
  5. esModuleInterop allows interoperability between commonjs and es modules and enables compatibility with Babel.
  6. forceConsistentCasingInFileNames to ensure consistent casing of imported files, no matter the system we work on
  7. moduleResolution set to “node”, to make sure that module dependencies get resolved in a way that mimics Node.js module resolution mechanism
  8. isolatedModules, to transpile (or type check in our case) each file as a separate module
  9. noEmit, to make sure TS doesn’t generate JS files from TS ones.
  10. jsx, to enable support typechecking for JSX in .tsx files
  11. include, to specify files and folders that TS will be type checking. We want the whole src folder to be type checked.

Configure Babel

Babel is a tool that converts new generation JS into an older version of JS. This is done so we can write more modern JS during development, and the be able to use that JS even in the older browsers that don’t support newer JS syntax. Or at least this is how I understand it.
With that said, we will use Babel to convert React and TS code to JS. Huh.

Ok, now we need to install Babel, as well as few other plugins that Babel will use to be able to its job:

  • @babel/core, which is a core Babel library
  • @babel/preset-env, which allows us to use latest JS features, while also supporting older browsers
  • @babel/preset-react, used to help Babel transform React code into JS
  • @babel/preset-typescript, same as above, but used for TS
  • @babel/plugin-transform-runtime and @babel/runtime, to make sure we can use async and await JS features

We will install these as dev dependencies, as we only need them during the development.

Configure Webpack

Now we configure Webpack.
Webpack will bundle all of our code into a single file that we will later use with index.html.
It will also have Babel transpile TS files, and we will set up Webpack to use few other plugins to deal with CSS files, type errors, development server and such.

We will start with basic packages and plugins:

  • webpack, the main tool that will bundle our code
  • webpack-cli, the command-line interface for webpack
  • webpack-dev-server, webpack’s development server
  • bable-loader, a tool used to transpile React and TS code to JS
  • ts-node, a Node module to make it possible for us to use TS

We will also install types for webpack, and webpack-dev-server, and we will install all of these as dev dependencies.

Next up, let’s created the config file. We will write it as a TS module, and the ts-node package that we installed will take care just make it work :).

Here is what the configuration file properties do:

  • entry - specifies where should Webpack start the process of bundling. In our case, src/index.tsx is the entry into our React app.
  • module - sets rules on how specific modules (files) should be dealt with. In our case, all .ts, .tsx, .js, and .jsx files should be processed by Babel loader
  • resolve.extensions - tells Webpack which file types should be look for in which order when resolving import statements.
  • output specifies where to output the bundled code. In our case, we want our code bundled into a bundle.js file, and stored in the build folder
  • devServer configures Webpack development server. We are setting the public folder as a directory from which static assets should be served (index.html), and using port 4000 for serving the app. I also added the historyApiFallback property as a solution to routing with React Router, encountered later in the series. Set to true, this will make all 404 errors fallback to index.html.

There are few other solutions to the routing issue here, including for possible problems in the future.

Back to the webpack configuration, note that we imported both webpack and webpackDevServer into the configuration file. Even though we are not actually using the latter one, both imports are needed for type-checking purposes.

We need to adjust the config file in few more places.

First, we want to add type-checking to webpack process. This way, if there is a type error in our code, Webpack will throw an error and inform us about it during the bundling process.
We do this with another plugin, for-ts-checker-webpack-plugin. We install it, together with its TS types, as a dev dependency.

Then we import it into the webpack configuration module, and add it to the plugins array by instantiating its constructor.
Two options, async and eslint, are used in the argument passed to the constructor. async set to false specifies that Webpack should actually wait for type-checking process to finish before it emits any code. eslint in its current form defines that Webpack should inform us of any linting errors in src folder, but only for .ts, .tsx, .js, and .jsx files.

We also want to make sure that our CSS files are handled by Webpack. We will set a rule that will translate CSS syntax to a format understandable to Webpack. The rule will check for .css extension, and use two loaders to deal with these files.

The loaders are css-loader and style-loader.
The first one just takes a CSS file and returns another one that includes the original CSS styles, toghether with resolved imports that Webpack can work with.
The second one takes actual CSS and inserts it into the page by creating style tag and placing the styles inside.
